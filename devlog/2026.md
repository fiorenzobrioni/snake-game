### 2026-06-21 - Step 6.9.13 per-skin item shapes (flat skins go square)

Made the on-board items speak each skin's own visual language instead of always being circular. The
discriminator is `SkinPalette.useGlow`: glow skins (Classic / Neon) keep their haloed **discs** (their
strength - left untouched), flat skins (Retro / Pixel) now render items as **rounded squares**.

- **Regular food.** The square branch in `drawFood` is now gated on `!useGlow` (was `cornerFactor < 0.06`,
  i.e. Pixel-only) so Retro gets squares too, and the corner is derived from the skin's `cornerFactor`
  (crisp Pixel, lightly rounded Retro). The grounding drop shadow follows the same square shape.
- **Specials (power-ups / hazards).** `drawSpecialFood` now takes the `palette` and branches: glow skins
  keep the haloed disc + circular dashed caution ring; flat skins draw a rounded square with a grounding
  shadow, a top sheen and a **square** dashed caution ring for hazards. The universal accent colour and the
  on-piece symbol are unchanged, so an effect still reads the same across skins.

No gameplay impact (pure rendering); the per-item pulse animation works unchanged on squares, and the
generic eat burst / shockwave effects were intentionally left as-is.

### 2026-06-21 - Fix stray lingering ring on the board

Bug from the juicier-particles work: an eat occasionally left a faint circle stuck on the board for a few
seconds. Cause: the eat "shockwave" ring is one shared `Animatable` keyed on the eat-event id; when an eat
was immediately followed by a food **vanishing**, the new `LaunchedEffect(eatEventId)` cancelled the
in-flight ring animation mid-value and the vanish branch never restarted it, leaving the ring frozen at a
mid value (so it kept drawing at the vanished food's cell until the next eat). Fix: always `snapTo(0f)`
at the start of every eat event before deciding whether to animate the ring, so a just-cancelled ring is
always cleared. No gameplay impact (the ring was purely cosmetic).

(Also: the short-lived "atmospheric background" experiment was reverted - the in-game board background is
back to the original gradient + two drifting glows + vignette, which reads cleaner during play.)

Hazards no longer strike unannounced: they get a one-tick "tell" plus a steady danger tag.

  **Reduce motion** (the pre-haptic still fires). Hazard specials also gained a steady dashed **caution
  ring** so a dangerous piece reads at a glance - the calm counterpart to the imminent flash. Wired

### 2026-06-20 - Daily Challenge

A date-seeded run that is the same for everyone on a given day, with a per-day best and a streak.

- **Model.** New pure-Kotlin `DailyChallenge.forDay(epochDay)` derives a SplitMix64 `seed` plus a fixed
  config (mode alternates Endless/Time Attack by day, level from a decorrelated hash, board pinned to
  Standard; Campaign is excluded). Clock-free and unit-tested (determinism, distribution, range).
- **Determinism.** The ViewModel reseeds a fresh `GameEngine(Random(seed))` for the run, so the obstacle
  layout and food sequence are the day's (for a given board size - the board stays responsive to the
  device aspect). The spawn-affecting toggles (`hazardsEnabled`, `specialFrequency`) are **pinned**
  during a daily so two players can't get different runs from their personal settings; the settings
  collector skips syncing the pinned fields and the daily mode/level/scale while a daily is configured.
- **Lifecycle.** `requestDailyStart` marks daily mode and defers the seeded start to the board
  measurement (like Quick Play). "Play again" replays the exact same seed; leaving via `toSetup` drops
  the seeded engine and restores the player's persisted Custom setup. Game-over routes the score to the
  per-day daily store (separate from the normal records) and advances the streak on the first play of a
  new day. HUD shows a "Daily" tag.
- **Persistence + UI.** `SettingsRepository` gains `dailyBest(epochDay)`, `submitDailyScore` (best +
  streak), `dailyStreak`, `dailyLastPlayedDay`. New `DailyChallengeScreen` (menu entry) shows today's
  config, best-today and the live streak (lapsed streaks read 0 until today is played) with a Play button.

### 2026-06-20 - Coyote / grace tick for kinder deaths

At speed a single mistimed input meant instant death. Now `GameEngine.tick` spends a banked "grace"
dodge on the first lethal step: the snake freezes in place for one tick (the head hesitates against the
hazard) instead of dying, buying a beat to turn away. The dodge is re-banked by the next safe move, so a
second lethal step with none left is still fatal - no permanent wall-riding. `GameState.graceAvailable`
defaults false (keeping the collision unit tests immediate) and is granted by `setup` / level staging.
A new `GameEvent.GraceDodge` drives a firmer "close call" haptic. Engine tests cover freeze-then-die,
the re-bank, and the setup grant.

### 2026-06-20 - Haptics and near-miss feedback

Tactile feedback for gameplay, mirroring the audio architecture.

- **`GameHaptics` sink + `HapticController`.** A new `GameHaptics` interface (audio package, parallel to
  `GameSfx`, with a `None` no-op for tests) is injected into `GameViewModel` alongside `sfx`. The
  production `HapticController` drives the device vibrator via `VibratorManager` + predefined
  `VibrationEffect`s (`minSdk 33` guarantees both): `eat` = TICK, `special` = CLICK, `lifeLost` =
  HEAVY_CLICK, `levelUp` = DOUBLE_CLICK, `death` = a short double-thud waveform, and `nearMiss` = a
  faint 10 ms one-shot. It observes `hapticsEnabled` live so the Settings toggle applies immediately;
  no vibrator → silent no-op. Added the `VIBRATE` permission and an `App`-owned instance (released on
  host destroy, like `GameAudio`).
- **Near-miss detection.** New `GameEvent.NearMiss`, emitted by `GameEngine.tick` when the surviving
  head is orthogonally adjacent to a **static** hazard (board edge, obstacle, level wall or debris).
  Self-body adjacency is excluded (coiling is not a graze) and it is skipped under Ghost. The
  ViewModel throttles the haptic to one every 220 ms so riding an edge gives an occasional tick rather
  than a continuous buzz.
- **Setting.** `Settings.hapticsEnabled` (default on) + a "Vibration feedback" toggle in Settings.

Verified: `./gradlew testDebugUnitTest` green (the existing event-matcher tests are robust to the extra
`NearMiss` events; no exact-list assertions). README updated.

### 2026-06-20 - Electric/plasma flow on the 3D barrier (perspective-mapped AGSL) + toggle

Added the optional luminous "electric" effect on the neon barrier the user asked for, with proper
perspective UV mapping (the harder but better-looking option) and a settings toggle (default on).

- **AGSL shader (`Shaders.WALL_FIELD`).** A subtle drifting fbm + thin travelling filaments that rise up
  the wall, brightest near the top/bottom rails, kept low-alpha for an elegant shimmer. It is fed a
  **screen->UV homography** so the energy flows *along the wall in perspective* rather than being pinned to
  the screen. `BoardShaders.wallField` is constructed with `runCatching` (nullable) so a compile failure on

- **Perspective UV (`GameBoard.kt`).** New `screenToUvHomography` (Heckbert square->quad, then `invert3x3`)
  maps a fragment's screen position to (u along the wall, v up it). It is applied only when the whole
  barrier quad is in front of the camera (`field.size == 4`); a run straddling the near plane falls back to
  the plain field for that frame, avoiding undefined projections. Layered over the existing translucent
  base, under the glowing rails.
- **Toggle.** New `electricWallsEnabled` setting (default **on**) in `SettingsRepository`, surfaced as an
  "Electric 3D walls" switch in Settings, threaded `GameViewModel.electricWalls` -> `GameScreen` ->
  `GameBoard(electricField=…)` -> `draw3DScene`. Geometry untouched.

Tuning follow-up (same day): the first pass was nearly invisible and grey. Made it markedly stronger
(crossing band sets + a constant coloured glow, higher alpha) and recoloured the field to the skin's vivid
accent (`palette.headGlow`: lime for Classic, cyan for Neon, ...) so it reads as electric/plasma instead of
grey; the structural rails stay border-neon.

### 2026-06-20 - 3D boundary redesigned as a luminous neon barrier (no thickness)

Per user direction, replaced the solid extruded arena walls with a **luminous, translucent neon barrier**
(`ui/game/GameBoard.kt`), fixing the remaining issues with the solid version: from some angles a wall
looked "scavato" (you saw into / past the faces), and the perpendicular boxes overlapped at the corners,
which was both ugly and flickered while moving.

- **No thickness.** Each arena side is now a single translucent "energy field" quad (the boundary line up

  Because the barrier has no depth: (1) it never hides the snake/food behind the back edges - you see

  z-fighting/flicker is gone; (3) a double-sided filled quad reads correctly from every angle (no hollow

- **Neon glow.** The (often dark) skin `boardBorder` is lifted to a vivid neon and the rails are drawn with
  a cheap layered bloom (wide soft halo + coloured core + hot near-white centre) via a new `glowLine`
  helper, with a gentle breathing pulse. The barrier fades/rises in with the tilt blend and collapses to
  the flat outline at blend 0. `WALL_HEIGHT` stays the single, easily-tweakable height knob.

The continuous-run machinery (`WallRun`, `wallRuns`, near-plane `clipPolygonNear`) from the previous step
is reused, so the barrier is still seam-free and gap-free. Verified: `./gradlew testDebugUnitTest` and
`./gradlew assembleDebug` green. README updated.

### 2026-06-20 - 3D walls rewritten as continuous strips (no break-up, uniform shade)

Follow-up on the near-plane fix below: the boundary walls still broke up. They were extruded **one box per
cell**, so adjacent boxes showed seams, per-cell depth-sort flicker, non-uniform shading and (from some
angles) a hollow look as the camera moved. Rewrote the wall rendering in `ui/game/GameBoard.kt`:

- **One continuous strip per wall.** Each arena side (and, in Levels mode, each straight stretch of the
  shaped boundary) is now a **single extruded box** - inner face, outer face, top cap and two end caps -
  instead of one box per cell. A planar quad projects exactly under perspective, so a single strip has no
  inter-cell seams and a uniform tone along its length. New `WallRun` type + `wallRuns()` builder: the
  rectangular board yields four runs (each extended by one wall thickness at both ends so the corners seal
  into solid posts); a shaped board merges the unit `boundaryEdges` into maximal collinear runs (same line
  + normal, contiguous).
- **Flat low-poly shading.** Faces use flat per-face tones (cap brightest, inner mid, outer/ends darker)
  with only a gentle ambient gradient, giving a clean, consistent read of a solid wall with real thickness.
  The five faces are near-clipped (`Cam.clipPolygonNear`) and depth-sorted among themselves so the box
  occludes itself; the bright inner top rail is kept (clipped). Removed the per-cell masonry seam lines
  that were producing the triangular artefacts, and the now-unused `WallFace`/`lerpVec` helpers.

Verified: `./gradlew testDebugUnitTest` green; `./gradlew assembleDebug` green.

### 2026-06-20 - 3D rendering consistency: gap-free grid, walls and floor (near-plane clipping)

Fixed the flickering artefacts in both 3D views (chase-cam and 3D Fixed) where, while moving, the floor
grid would partly vanish (leaving only the lines perpendicular to travel) and the boundary walls would
break into gaps. Root cause: geometry crossing the camera's **near plane** was *culled wholesale* instead
of being *clipped* to it.

- **Near-plane projection round-off (vanishing grid lines).** `Cam.clipNear` trimmed a segment to exactly
  `z = NEAR`, but `Cam.projectCamera` then rejected it with `z <= NEAR`, so the line it had just clipped
  was dropped. Grid lines running along the direction of travel cross the near plane (lines across it do
  not), which is why only the perpendicular lines survived. Added a `NEAR_CLIP` plane a hair in front of
  `NEAR` and clip to it, so clipped vertices land strictly in front of `NEAR` and pass the visibility test
  (`ui/game/ChaseCam.kt`).
- **Whole-segment wall culling (gaps in the walls).** `addWall` dropped an entire extruded wall segment if
  *any* of its 8 corners was behind the near plane, punching holes in the arena wall as the camera swept
  along it. It now clips each face polygon to the near plane via the new `Cam.clipPolygonNear`
  (Sutherland-Hodgman against one plane), so a straddling cell is **trimmed, not dropped**. Masonry seams
  and the inner top rail are clipped per-line from their original edge corners (new `WallFace` carrier +
  `lerpVec` helper) so they stay aligned even when the face is trimmed. The depth-sort key is recomputed
  from the surviving geometry (`ui/game/GameBoard.kt`).
- **Floor flicker.** The solid floor quad was drawn only when *all four* corners were visible, so it
  blinked off (revealing the bare grid) whenever a corner passed behind the camera. It is now near-clipped
  too, staying a continuous filled surface.

Discrete objects (obstacles, food, snake segments, particles) keep their existing per-object near cull -
an item passing directly under the camera disappearing reads as natural, unlike a gap in a continuous
structure.

### 2026-06-05 - Feedback round 5: first-run board overflow - ACTUAL root cause

- It was **not** an inset-timing issue (round 4 was a red herring, though the inset cleanup is kept).
  The board-shake offset was non-zero **at rest**: `shakeY` used `cos(shakeT·…)·amp·damp`, and at idle
  `shakeT=0` → `cos(0)=1`, `damp=1`, so the board sat ~`amplitude (10dp) + quakeAmplitude (7dp)` ≈ 17dp
  **down** before any shake. Its bottom (and the snake) ran off-screen on the first game; the first
  death animation drove `shakeT→1` (`damp=0`), removing the 10dp term so later games looked fine
  (~7dp, absorbed by the board margin). Only the bottom was affected because X used `sin` (0 at rest)
  while Y used `cos`; Cozy overflowed less because its shorter board leaves more bottom margin.
- Fix: use `sin` on both axes so the offset is exactly 0 at rest (and still a proper damped wobble
  during a shake). The stray 7dp came from the quake term added in Phase 6.

### 2026-06-05 - Feedback round 4: first-run board overflow (inset cleanup)

- The board ran under the navigation bar on the **first** game only (snake's bottom row half off-screen;
  worse with denser scales, barely with Cozy) and corrected from the second game. Root cause: the
  safe-area inset padding (`safeDrawingPadding`) was applied **inside** the `Crossfade`, so the first
  `GameScreen` created a fresh padding node that briefly saw 0 insets - the play area was measured at
  full window height (under the nav bar) and the board dimensions were locked to it.

  persistent node), so the play area is always inset-correct when GameScreen first appears. Belt-and-
