### 2026-06-26 - More tuning: symmetric Level 11, respawn explosion, 4 foods on big boards

- **Level 11 (the Lattice) made symmetric.** A single centre-anchored periodic grid can't stay
  mirror-symmetric on even board dimensions, so it looked lopsided. Rebuilt by tiling the top-left
  quadrant and mirroring every pillar into the other three quadrants, giving a 4-fold symmetric field
  on any board size while keeping the same sparse, staggered difficulty. (`game/LevelsMode.kt`.)

- **Death explosion now also plays on a non-final Campaign death.** Previously only the final death
  (game over) burst the snake apart; losing a life mid-run respawned instantly. The level-intro
  transition now bursts the old snake (the death explosion) before a respawn too, alongside the
  existing teleport-vanish for a level-up. Reduce-motion still stages instantly. (`ui/game/
  GameViewModel.kt`.)

- **Level-up teleport lengthened** from 760 to 1000 ms, matching the death animation; the renderer's
  dissolve window is 900 ms (just under both holds). (`ui/game/GameViewModel.kt`, `GameBoard.kt`.)

- **Big boards carry 4 simultaneous foods.** `GameEngine.foodCountFor(board)` returns 4 when the
  board's short side is >= 24 cells (Epic / Colossal), else the usual 3, so the larger arenas don't
  feel empty. Used at both refill sites; `FOOD_COUNT` stays the base of 3. (`game/GameEngine.kt`.)

Verified: `./gradlew testDebugUnitTest assembleDebug lintDebug` all pass (incl. the LevelShapes
connectivity / 60%-playable guards for the new Lattice).

### 2026-06-26 - Tuning pass: Level 11, start length 4, longer transitions

- **Level 11 (the Lattice) loosened further.** Widened the pillar spacing from every 4 to every 5
  columns (still alternate rows, still offset), dropping density to ~1 pillar per 10 cells. Clearly
  hard but no longer punishing; pillars stay isolated so connectivity holds. (`game/LevelsMode.kt`.)

- **Snake spawns at 4 segments** (was 3). New `GameEngine.START_LENGTH = 4`; `startingSnake` builds it
  with `List(START_LENGTH)`. The shrink floor stays `MIN_SNAKE_LENGTH = 3`, so a snake can still be
  shrunk down to 3. The centre spawn fits the protected zone (`cy-4..cy+3`) on every board, and
  obstacle/food placement already avoids the snake cells. Updated the spawn-length unit test.
  (`game/GameEngine.kt`, `GameEngineTest.kt`.)

- **Longer transition animations.** Death burst `DEATH_ANIM_MS` 720 -> 1000 ms and the level-up vanish
  `LEVEL_VANISH_MS` 520 -> 760 ms, with the renderer's dissolve windows bumped to match (900 / 680 ms,
  kept just under each hold so the body has faded before the overlay / countdown). (`ui/game/
  GameViewModel.kt`, `GameBoard.kt`.)

- **Checked: snake speed is independent of length.** `GameState.tickIntervalMillis` depends only on
  mode, elapsed time (Endless), speed cycle (Levels), the chosen speed and power-ups - never on length;
  the snake always advances one cell per tick. (Endless even quickens over time, so a longer snake is
  if anything faster.) The "short snake feels faster" impression is a visual illusion, not a bug; no
  change made.

### 2026-06-26 - Fix invisible-snake regression + Campaign level names

Follow-up to the 7.8 polish batch (PR #49).

- **Fix: snake invisible (only the eyes) on subsequent games.** The transition animations
  left the renderer's `dissolve` envelope stuck at 0. Starting a fresh game - especially via
  Menu -> Game, which recomposes `GameBoard` - re-ran `LaunchedEffect(bodyBurstId)` with the
  *leftover* `bodyBurst`/`bodyBurstId` from the previous run, replaying the burst on the new
  snake and fading it to invisible (the eyes stayed because `drawEyes` ignored the head
  alpha). Guarded the effect with a remembered `handledBurst` id initialised to the current
  `bodyBurstId`, so a fresh composition is a no-op; only a genuinely new id runs a burst. Also
  threaded the head `alpha` into `drawEyes` so the eyes fade with the head during the
  death/level-up dissolve. (`ui/game/GameBoard.kt`.)

- **Campaign level names on the intro screen.** The 15 designed level names lived only in
  KDoc; promoted them to data via `LevelsMode.nameFor(levelIndex)` (wraps per cycle) and the
  level-intro overlay now shows the name (e.g. "The Lattice") between "Level X" and "Speed Y",
  styled like the Speed line in the tertiary accent. (`game/LevelsMode.kt`,
  `ui/game/GameOverlays.kt`, `ui/game/GameScreen.kt`.)

Verified: `./gradlew testDebugUnitTest assembleDebug lintDebug` all pass.

### 2026-06-26 - Premium polish: Daily replay, transition animations, Level 11, CRT, debug unlock

A five-part polish batch keeping the premium bar.

- **Replay a past Daily Challenge** from the weekly history (`ui/daily/DailyHistoryScreen.kt`). The
  per-day rows are now tappable (no layout change) and open a confirmation dialog ("Replay this day?");
  confirming relives that day's exact seeded challenge. A replay leaves `dailyEpochDay` null, so it runs
  through the existing not-recorded path (`GameViewModel.requestReplayStart` / `replayDay`): the day's

- **Transition animations** (`ui/game/GameViewModel.kt`, `GameBoard.kt`, `GameScreen.kt`). On death the
  snake now bursts apart into particles and fades out before the game-over overlay (held back by
  `deathAnimating`); on a Campaign level-up the completing snake dissolves away with a teleport-style
  sparkle before the next-level countdown (a brief `levelVanishing` hold with the loop stopped so the
  old snake isn't ticked again). Both reuse the existing particle bursts via a new whole-snake
  `BodyBurstEvent` channel and a `dissolve` envelope in the renderer; the board's effects loop is kept
  alive across the transition via a new `effectsActive` flag. Fully suppressed under reduce-motion
  (instant behaviour, as before); Campaign respawn-after-life-lost stays instant too.

- **Level 11 (the Lattice) rebalanced** (`game/LevelsMode.kt`). The quincunx put a pillar every four
  columns on *every* row (~25% density); it now skips every other row (pillars only on alternate rows,

  the Colonnade; pillars stay isolated so connectivity holds.

- **CRT filter made more incisive** (`ui/game/Shaders.kt`). The old scanlines used a resolution-scaled
  frequency so high it aliased into near-invisibility. Switched to a fixed ~3px pixel period with a
  stronger amplitude, added a faint aperture-grille shimmer, and deepened the vignette.

- **Debug-only "unlock all themes" button** (`ui/menu/MainMenuScreen.kt`, `app/build.gradle.kts`).
  Enabled `buildConfig`; a `BuildConfig.DEBUG`-gated icon button (left of Settings) unlocks every skin

### 2026-06-23 - Onboarding redesign: dark/minimal, detailed legends, real special icons

A premium pass over the first-run tutorial (`ui/onboarding/OnboardingScreen`), prompted by three issues: the
artwork inherited each skin's (sometimes bright/warm) board so it did not feel dark and minimal; the copy was
thin; and the last card's specials were a single flat colour with no symbols, and clipped off-screen because the
card was a centred, non-scrolling column.

- **Dark, minimal backdrop.** The screen now owns its own full-bleed `AnimatedShaderBackground` (the same
  drifting-glows AGSL shader as the intro / menus), so it always reads dark regardless of theme. `App` excludes
  `Screen.Onboarding` from the shared backdrop to avoid stacking two animated shaders. The per-card illustration
  sits on a new neutral **dark mini-board** (`drawMinimalBoard`: cool gradient, faint grid, vignette, a soft
  skin-tinted rim) instead of the skin's own board, so the bright skin-coloured pieces pop. Text uses fixed light
  colours and titles take the skin's bright head colour, so contrast holds on any skin/theme.

- **No more clipping.** Each card is now a `verticalScroll` column, so the richer copy and legends never clip on
  short screens (the root cause of the "specials not visible" report).

- **Detailed legends.** New shared `InfoRow` (badge + bold accent title + description) drives three legend lists:
  the **Controls** card explains all three schemes (Swipe / Tap-to-turn / D-pad) with drawn glyph badges; the
  **Food** card shows Grow / Shrink / Mystery as real food pieces; the **Power-ups & hazards** card lists each
  special with its real in-game disc, accent colour and a one-line description, grouped under Power-ups and
  (red-ringed) Hazards. Copy in `strings.xml` was expanded throughout.

- **Real special icons (shared).** The special on-disc symbols (bolt, spiral, star, snowflake, "$", burst,
  crack, clock, snake-head) and `drawGlyph` were extracted from `GameBoard` into a new `ui/game/SpecialIcons.kt`
  (`internal`), so the tutorial and the live board render specials from a single source of truth - a colour and a
  symbol always mean the same thing. `GameBoard` now calls the shared helpers.

### 2026-06-21 - Step 6.9.7 Campaign environmental hazards: moving-wall gates & teleport portals

Brought the Campaign (`GameMode.Levels`) boards to life with two timed hazards layered on top of the
existing procedural wall shapes.

- **Model (`game/LevelHazards.kt`, new).** `Gate` is a group of cells with a deterministic, tick-phased
  open/close cycle (`period`/`openTicks`/`offsetTicks`, all read off the run's monotonic `elapsedTicks`),
  plus `isClosingSoonAt` / `isOpeningSoonAt` warning windows for the renderer. `TeleportPair` maps each pad
  to its partner (`exitFor`). `LevelHazards` bundles the two lists.
- **Assignment & safety (`LevelsMode.hazardsFor`).** Designed hazards per level: teleport portals on Open
  Field (two crossed pairs), Hourglass (a waist shortcut) and the Vault (inside↔outside); moving-wall gates

### 2026-06-20 - 3D rendering consistency: gap-free grid, walls and floor (near-plane clipping)

Fixed the flickering artefacts in both 3D views (chase-cam and 3D Fixed) where, while moving, the floor
grid would partly vanish (leaving only the lines perpendicular to travel) and the boundary walls would
break into gaps. Root cause: geometry crossing the camera's **near plane** was *culled wholesale* instead
of being *clipped* to it.

- **Near-plane projection round-off (vanishing grid lines).** `Cam.clipNear` trimmed a segment to exactly
  `z = NEAR`, but `Cam.projectCamera` then rejected it with `z <= NEAR`, so the line it had just clipped
  was dropped. Grid lines running along the direction of travel cross the near plane (lines across it do
  not), which is why only the perpendicular lines survived. Added a `NEAR_CLIP` plane a hair in front of
  `NEAR` and clip to it, so clipped vertices land strictly in front of `NEAR` and pass the visibility test
  (`ui/game/ChaseCam.kt`).
- **Whole-segment wall culling (gaps in the walls).** `addWall` dropped an entire extruded wall segment if
  *any* of its 8 corners was behind the near plane, punching holes in the arena wall as the camera swept
  along it. It now clips each face polygon to the near plane via the new `Cam.clipPolygonNear`
  (Sutherland-Hodgman against one plane), so a straddling cell is **trimmed, not dropped**. Masonry seams
  and the inner top rail are clipped per-line from their original edge corners (new `WallFace` carrier +
  `lerpVec` helper) so they stay aligned even when the face is trimmed. The depth-sort key is recomputed
  from the surviving geometry (`ui/game/GameBoard.kt`).
- **Floor flicker.** The solid floor quad was drawn only when *all four* corners were visible, so it
  blinked off (revealing the bare grid) whenever a corner passed behind the camera. It is now near-clipped
  too, staying a continuous filled surface.

Discrete objects (obstacles, food, snake segments, particles) keep their existing per-object near cull -
an item passing directly under the camera disappearing reads as natural, unlike a gap in a continuous
structure.

Verified: `./gradlew testDebugUnitTest` green (incl. `ChaseCamTest`); `./gradlew compileDebugKotlin` clean.

### 2026-06-05 - Feedback round 5: first-run board overflow - ACTUAL root cause

- It was **not** an inset-timing issue (round 4 was a red herring, though the inset cleanup is kept).
  The board-shake offset was non-zero **at rest**: `shakeY` used `cos(shakeT·…)·amp·damp`, and at idle
  `shakeT=0` → `cos(0)=1`, `damp=1`, so the board sat ~`amplitude (10dp) + quakeAmplitude (7dp)` ≈ 17dp
  **down** before any shake. Its bottom (and the snake) ran off-screen on the first game; the first
  death animation drove `shakeT→1` (`damp=0`), removing the 10dp term so later games looked fine
  (~7dp, absorbed by the board margin). Only the bottom was affected because X used `sin` (0 at rest)
  while Y used `cos`; Cozy overflowed less because its shorter board leaves more bottom margin.
- Fix: use `sin` on both axes so the offset is exactly 0 at rest (and still a proper damped wobble
  during a shake). The stray 7dp came from the quake term added in Phase 6.

### 2026-06-05 - Feedback round 4: first-run board overflow (inset cleanup)

- The board ran under the navigation bar on the **first** game only (snake's bottom row half off-screen;
  worse with denser scales, barely with Cozy) and corrected from the second game. Root cause: the
  safe-area inset padding (`safeDrawingPadding`) was applied **inside** the `Crossfade`, so the first
  `GameScreen` created a fresh padding node that briefly saw 0 insets - the play area was measured at
  full window height (under the nav bar) and the board dimensions were locked to it.

  persistent node), so the play area is always inset-correct when GameScreen first appears. Belt-and-
  suspenders: `GameViewModel.start()` re-fits the board to the latest measured area before locking it.
